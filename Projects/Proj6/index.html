The Analysis And Design Phase Of A Program

(Make Sure Your Internet Connection is Working)

The Following Are The Six Genral Phases of the Software Development Process:

  1. Analysis
  2. Design
  3. Implementation or coding
  4. Testing
  5. Deployment
  6. Maintenance

This Article Will Mostly Deal with the first two of the list.

Analysis Phase

The purpose of the analysis phase is to determine where the problem that the software project is supposed to address or fix. This step involves breaking down the project into different pieces to analyze the situation, analyze project goals, break down what needs to be created and attempting to engage users so that definite requirements can be defined. This may also give answer to questions like what tools and technologies should be or is possible to use, what can be the project schedule, make better budget estimations, etc.During the the analysis a requirement documentshould be created. This document should state in a clear and precise fashion what is to be built.

Analysis Phase: In Short

  • Gather information
  • Define system requirements
  • Generate and evaluate alternatives
  • Prioritise requirements
  • Prototype for feasibility and discovery
  • Review recommendations with management

Design Phase

The purpose of the design phase is to develop a specification for a program or piece of software that will meet your project needs. During this phase it is important take into account the time, resources, and cost of building your project. During this phase, your consultants assemble the information they gathered from you during the analysis phase.

During the Design phase you establish best practices to follow for this phase by way of functional and design specifications, and perform risk analysis to identify threats and vulnerabilities in your software.

During this phase visual components of the project will be created. This includes actual layouts,flowcharts, diagrams, pseudocode and other visual documentation.

Design Phase

A design pattern is generally described as a reusable solution to a commonly occurring problem within a given context in software design. A design pattern is not a finished design that can be transformed directly into source code. It is a description or template for how to solve a problem that can be used in many different situations. Patterns are formalized best practices that programmers can implement in the application. Object-oriented design patterns typically show relationships and interactions between classes or objects, without specifying the final application classes or objects that are involved.
